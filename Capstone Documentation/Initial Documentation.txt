AI Tool Usage Documentation
M365 Copilot and ChatGPT were used as supportive learning and productivity tools throughout the development of this project. All final design decisions, implementations, and integrations were completed by me after reviewing and understanding the generated suggestions.
M365 Copilot was used during the initial setup of the Spring Boot backend to assist with project configuration in IntelliJ IDEA, including generating a basic application structure and verifying that the Spring Boot application was running correctly. A simple HealthController was created to confirm successful startup and routing.
M365 Copilot was also used to assist with configuring application.properties and identifying appropriate dependencies for inclusion in pom.xml. All suggested configurations were reviewed, modified where necessary, and manually integrated to align with the project’s requirements.
ChatGPT was used primarily as a learning aid during the setup of the React frontend and while troubleshooting backend issues, including JPA entity mappings, REST endpoint behavior, and date/time handling with MySQL. These interactions were used to better understand concepts and resolve errors; all final implementations were written and refined by me.
AI tools were also used to generate synthetic mock and dummy data for the MySQL database tables (venues, events, tickets, users, and ticket sales). This data was used exclusively for development and testing purposes, including pagination, filtering, analytics, and edge-case validation. The generated data was reviewed and adjusted as needed before being integrated into the project.
The database schema was initially exported from DrawSQL and later refactored to follow SQL and Java naming conventions (snake_case, lowercase) to ensure compatibility with MySQL and Spring Boot JPA. This refactor did not change any functional behavior.
Overall, AI tools were used to support learning, validation, and efficiency, not to replace independent implementation or decision-making.
M365 Copilot was given the original documentation, to polish it. M365 Copilot was used in creating mocking data for the database "capstone_improved". A combination of M365 copilot and the website drawSQL was used to setup and polish the initial ERD and database tables. The design was created within drawSQL then exported.
M365 was used to help setup the dev branch on GitHub.

Frontend Setup (React + NPM Follow‑Along)
This section documents the steps used to initialize and prepare the React frontend for the capstone project.
Prerequisites

Node.js installed
NPM available from the command line


Step 1: Create React Application Using Vite
The React application was scaffolded using Vite, a modern frontend build tool.
From the project directory, run:
Shell: npm create vite@latest practice-capstone-frontend
When prompted, select:

Framework: React
Variant: JavaScript

After creation, navigate into the project and install dependencies:
cd practice-capstone-frontend
npm install
npm run dev

This starts the development server and confirms the React environment is working.

Step 2: Clean Default Boilerplate
Default files and assets generated by Vite that were not required for the capstone were removed to keep the project clean and understandable.
Deleted:

src/assets/ (default images)
src/App.css (unused default styling)

Optional removals:

README.md
eslint.config.js

Core files retained:

index.html
vite.config.js
package.json
src/main.jsx
src/App.jsx


Step 3: Verify Basic Rendering
App.jsx was simplified to confirm the application renders correctly:
JSXfunction App() {  return <h1>Practice Capstone</h1>;}export default App;Show more lines
Running npm run dev at this stage displays the header text in the browser, confirming successful setup.

Step 4: Create Project Folder Structure
A scalable folder structure was created inside src/ to support future development:
Plain Textsrc/ ├── api/ ├── components/ ├── pages/ ├── services/ ├── utils/ ├── App.jsx └── main.jsxShow more lines
This structure separates concerns and supports clean integration with the backend later in the project.

Step 5: Set Up Routing
Client-side routing was added using React Router.
Install dependency:
Shellnpm install react-router-domShow more lines
Basic routing was configured in App.jsx to support multiple pages.

Step 6: Configure Environment Variables
To prepare for backend integration, an environment variable was created for the API base URL.
Create a .env file in the project root:
Plain Textenv isn’t fully supported. Syntax highlighting is based on Plain Text.VITE_API_URL=http://localhost:8080/apiShow more lines
This allows the frontend to connect to the Spring Boot backend without hardcoding URLs.

Step 7: Prepare API Layer
An Axios instance was created to centralize API communication:
JavaScriptimport axios from "axios";const api = axios.create({  baseURL: import.meta.env.VITE_API_URL,});export default api;Show more lines
This file will be reused when backend endpoints are implemented.

Notes

TypeScript was intentionally not used to reduce complexity during a time‑limited capstone.
UI styling and API calls were deferred until backend development was completed.
This setup ensures the frontend is backend‑ready without unnecessary overhead.
