Initial endpoints implemented for Events. Additional endpoints for Users, Venues, and Bookings will follow the same REST pattern.

Get an event by ID (no pagination):
http://localhost:8080/api/events/id/{id}

Get all events (pagination supported):
http://localhost:8080/api/events
Example:
http://localhost:8080/api/events?page=0&size=5&sortBy=date&ascending=false

Get events by status (pagination supported):
http://localhost:8080/api/events/status/{COMPLETED | CANCELLED | SCHEDULED}
Example:
http://localhost:8080/api/events/status/cancelled?page=0&size=5

Get events between two dates (pagination supported):
http://localhost:8080/api/events/date?start=2020-01-07&end=2020-02-08
Example:
http://localhost:8080/api/events/date?start=2020-01-07&end=2020-02-08&page=0


Create a new event
POST http://localhost:8080/api/events
- Creates a new event
- Input is validated
- Event ID is generated automatically
- Returns the created event with HTTP 201 (Created)


Update event status (status-only modification)
PUT http://localhost:8080/api/events/id/{id}/{status}

Example:
PUT http://localhost:8080/api/events/id/1044/cancelled
- Updates only the event status
- Intended for administrative actions (e.g. cancelling an event)
- Event date and venue are intentionally immutable to preserve historical - accuracy
- Events are not deleted; cancelled events remain available for reporting and auditing

## **Venues API Endpoints**

Get a venue by ID (no pagination):

    http://localhost:8080/api/venues/id/{id}

*   Returns a single venue by its ID
*   Returns 200 OK if the venue exists
*   Returns 404 Not Found if the venue does not exist

Example:

    http://localhost:8080/api/venues/id/1

***

Get all venues (pagination supported):

    http://localhost:8080/api/venues

Query parameters (all optional):

*   `page` (default: 0)
*   `size` (default: 10)
*   `sortBy` (default: id)
*   `ascending` (default: true)
*   `city`
*   `state`

Description:

*   Returns a paginated list of venues
*   Supports sorting using Java entity field names
*   Supports filtering by city and/or state

Examples:

    http://localhost:8080/api/venues?page=0&size=5
    http://localhost:8080/api/venues?city=Columbus
    http://localhost:8080/api/venues?state=OH

***

Create a new venue:

    POST http://localhost:8080/api/venues

*   Creates a new venue
*   Input is validated
*   Venue ID is generated automatically
*   Returns the created venue with HTTP 201 (Created)

Example request body:

    {
      "venue_name": "Nationwide Arena",
      "location": "Columbus, OH",
      "total_capacity": 20000
    }

***

Update venue location (location‑only modification):

    PUT http://localhost:8080/api/venues/id/{id}/{location}

*   Updates only the venue location
*   Intended for administrative corrections
*   Other venue fields remain unchanged
*   Returns 200 OK

Example:

    http://localhost:8080/api/venues/id/1/Los%20Angeles

***

Get venues by location (legacy endpoint – not recommended):

    http://localhost:8080/api/venues/location/{location}

*   Returns venues filtered by full location string
*   Pagination and sorting supported
*   This endpoint is maintained for compatibility, but query‑parameter filtering (`city` / `state`) is preferred

Example:

    http://localhost:8080/api/venues/location/Columbus,%20OH?page=0&size=5

***

Notes (Venues):

*   Venues can be created, retrieved, and updated (location only).
*   Venues are not deleted.
*   Pagination and sorting are supported.
*   Query‑parameter filtering by city/state is the preferred approach.

## **Tickets API Endpoints**

Get a ticket by ID (no pagination):

    http://localhost:8080/api/tickets/id/{id}

*   Returns a single ticket by its ID
*   Returns 200 OK if the ticket exists
*   Returns 404 Not Found if the ticket does not exist

Example:

    http://localhost:8080/api/tickets/id/1

***

Get all tickets (pagination supported):

    http://localhost:8080/api/tickets

Query parameters (all optional):

*   `page` (default: 0)
*   `size` (default: 10)
*   `sortBy` (default: id)
*   `ascending` (default: true)
*   `minPrice`
*   `maxPrice`

Description:

*   Returns a paginated list of tickets
*   Supports optional price range filtering
*   Sorting uses Java entity field names

Examples:

    http://localhost:8080/api/tickets?page=0&size=5
    http://localhost:8080/api/tickets?minPrice=20.00
    http://localhost:8080/api/tickets?maxPrice=50.00

***

Get tickets by event ID (pagination supported):

    http://localhost:8080/api/tickets/event/{event_id}

*   Returns the ticket record associated with a specific event
*   Each event has at most one ticket record
*   Pagination metadata is included in the response

Example:

    http://localhost:8080/api/tickets/event/1?page=0&size=5

***

Create a new ticket:

    POST http://localhost:8080/api/tickets

*   Creates a ticket record for an event
*   Each event may have only one ticket record (enforced by a unique constraint on `event_id`)
*   Input is validated
*   Ticket ID is generated automatically
*   Returns the created ticket with HTTP 201 (Created)

Example request body:

    {
      "event_id": 1,
      "price": 48.86,
      "total_quantity": 100,
      "sold": 10
    }

***

Update ticket sold count (sold‑only modification):

    PUT http://localhost:8080/api/tickets/id/{id}/sold/{sold}

*   Updates only the number of tickets sold
*   Intended for administrative updates
*   Other ticket fields remain unchanged
*   Returns 200 OK

Example:

    http://localhost:8080/api/tickets/id/1/sold/50

***

Delete a ticket:

    DELETE http://localhost:8080/api/tickets/delete/{id}

*   Deletes a ticket record by ID
*   Intended for administrative cleanup
*   Returns HTTP 204 (No Content)

Example:

    http://localhost:8080/api/tickets/delete/1

***

Notes (Tickets):

*   Tickets represent ticket inventory for an event, not individual purchases.
*   Each event may have only one ticket record.
*   Pagination and sorting are supported on list endpoints.
*   Tickets may be deleted via an administrative endpoint.

***




Users API Endpoints
Get a user by ID (no pagination):
http://localhost:8080/api/users/id/{id}
Description:

Returns a single user by their unique ID
Returns 200 OK if the user exists
Returns 404 Not Found if the user does not exist

Example:
http://localhost:8080/api/users/id/1

Get all users (pagination supported):
http://localhost:8080/api/users
Query parameters (all optional):
page (default: 0)
size (default: 10)
sortBy (default: id)
ascending (default: true)
Description:

Returns a paginated list of users
Sorting is supported using Java entity field names
Valid sortBy values include id, username, and role
Invalid sortBy values fall back to sorting by id

Examples:
http://localhost:8080/api/users?page=0&size=10
http://localhost:8080/api/users?page=0&size=10&sortBy=username&ascending=true
http://localhost:8080/api/users?page=0&size=10&sortBy=role&ascending=true
http://localhost:8080/api/users?page=0&size=10&sortBy=username&ascending=false

Create a new user (registration):
POST http://localhost:8080/api/users
Description:

Creates a new user account
Request body is validated
User ID is generated automatically
Returns the created user with HTTP 201 (Created)

Example request body:
{
"username": "admin",
"password": "adminpass",
"role": "ADMIN"
}

Error handling behavior (Users):

Invalid request data returns HTTP 400 (Bad Request)
Requesting a non‑existent user ID returns HTTP 404 (Not Found)
Invalid sortBy parameters do not crash the API and fall back to default sorting


Notes (Users):

Users represent authenticated accounts in the system
Roles are limited to USER and ADMIN
User deletion is intentionally not supported to preserve historical integrity
Users may be referenced by ticket purchase records
Authentication and authorization will build on this user model

Tickets Sold API Endpoints
Get a ticket sale by ID (no pagination):
http://localhost:8080/api/tickets-sold/id/{id}
Description:

Returns a single ticket purchase record by its unique ID
Returns 200 OK if the record exists
Returns 404 Not Found if the record does not exist

Example:
http://localhost:8080/api/tickets-sold/id/1

Create a ticket sale record:
POST http://localhost:8080/api/tickets-sold
Description:

Creates a new ticket purchase record
Represents a user purchasing a ticket for an event
Request body is validated
TicketSold ID is generated automatically
Returns the created record with HTTP 201 (Created)

Example request body:
{
"user_id": 3,
"ticket_id": 12,
"date_sold": "2019-09-27"
}

Get all ticket sales (pagination supported):
http://localhost:8080/api/tickets-sold
Query parameters (all optional):
page (default: 0)
size (default: 10)
sortBy (default: id)
ascending (default: true)
month
year
Description:

Returns a paginated list of ticket purchase records
Supports sorting using Java entity field names (id, user_id, ticket_id, dateSold)
Supports optional filtering by month and/or year of purchase
When no filters are provided, all ticket sales are returned

Examples (pagination only):
http://localhost:8080/api/tickets-sold?page=0&size=10

Search ticket sales by month:
http://localhost:8080/api/tickets-sold?month=4
Description:

Returns all ticket sales that occurred in the specified month
Month values range from 1 (January) to 12 (December)

Example:
http://localhost:8080/api/tickets-sold?month=4

Search ticket sales by year:
http://localhost:8080/api/tickets-sold?year=2020
Description:

Returns all ticket sales that occurred in the specified year

Example:
http://localhost:8080/api/tickets-sold?year=2020

Search ticket sales by month and year:
http://localhost:8080/api/tickets-sold?month=4&year=2020
Description:

Returns all ticket sales that occurred in the specified month and year

Example:
http://localhost:8080/api/tickets-sold?month=4&year=2020

Search with pagination and sorting combined:
http://localhost:8080/api/tickets-sold?month=4&year=2020&page=0&size=5&sortBy=dateSold&ascending=false
Description:

Returns the first page of ticket sales for April 2020
Results are sorted by dateSold in descending order


Error handling behavior (Tickets Sold):

Invalid request data returns HTTP 400 (Bad Request)
Requesting a non‑existent ticket sale ID returns HTTP 404 (Not Found)
Invalid sortBy values do not crash the API and fall back to default sorting


Notes (Tickets Sold):

TicketSold records represent completed ticket purchases
Each record links a user to a ticket and records the purchase date
TicketSold records are not deleted to preserve historical purchase data
Month and year filtering supports reporting and administrative views
